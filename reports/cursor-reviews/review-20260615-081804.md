## Findings — ranked, top 10 (all NEW)

### 1. HIGH — REQ-114's headline "ONE hard budget ceiling" does not hold for the first (or any single) step
The run-mode gate never uses the per-step cost estimate. It admits each step against the **max actual cost of prior steps**, or `_MIN_STEP_EST_USD = 0.001` when there is no prior:

```563:571:src/rondo/pipeline.py
    for step in spec.steps:
        est = high_cost if high_cost > 0 else _MIN_STEP_EST_USD
        if spent + est > spec.budget_usd:
            status = "partial"
            pipeline_error = (
                f"ERR_BUDGET_EXCEEDED: spent ${spent:.4f} + estimated ${est:.4f} for step "
```

For step 0, `high_cost == 0`, so `est == 0.001`. A pipeline with `budget_usd: 0.50` whose only step calls an expensive model and actually costs $5.00 is admitted (`0 + 0.001 > 0.50` is false), dispatches, and spends $5.00 — a 10x breach of the "hard ceiling." There is **no post-dispatch check** that `spent <= budget`, so a single step can overshoot by its full actual cost, unbounded. The model-aware estimator that *would* catch this exists but is only ever called in plan mode (`_estimate_step_cost` → `_build_plan`, `pipeline.py:300`; confirmed it has no other caller). So `--plan` can print `within_budget_estimate: false` for that same pipeline and `apply` will still run it. The run envelope doesn't even include `budget_usd`, so the breach is invisible in the output.

Spec: req 010 (MUST) "`budget_usd` is a HARD pipeline ceiling" (`Rondo-REQ-114-prompt-pipelines.md:62`) and §4 "the hard ceiling is enforced on actuals as they land" (line 112) — both false for the first step and for any single-step overshoot. The existing tests bake the gap in as correct: `test_r010` uses three equal $0.6 steps under a $1 budget (trips on step 3), and `test_pipeline_mutation_kills.py::test_budget_gate_uses_prior_step_high_cost` pins the prior-step mechanism — neither tests a single expensive step against a small budget. This is the biggest correctness/overclaim gap in the thesis feature.

### 2. HIGH-MED — req 023 lists per-step `duration` as a MUST; it is never recorded and never tested
The step record is built without any timing, and the envelope adds none:

```492:492:src/rondo/pipeline.py
    record: dict[str, Any] = {"name": step.name, "status": "error", "raw_output": "", "cost_usd": 0.0, "error": ""}
```

`grep duration` over `src/rondo/pipeline.py` and over every `*pipeline*` test returns nothing. A MUST-listed envelope field is simply absent, and no conformance test would catch its absence — a clean claimed-built-but-unbuilt-and-untested req.

### 3. MED — the `passed=false` gate hijacks the reserved key from any step's legitimate JSON output, with no opt-out
`_self_reported_failure` is run unconditionally for every step (`pipeline.py:469`), and it fails the step if *any* JSON object anywhere in the output has `passed is False`:

```415:419:src/rondo/pipeline.py
    for parsed in _all_json_objects(raw_output):
        if parsed.get("passed") is False:
            detail = parsed.get("issues") or parsed.get("result") or "no detail given"
            return f"ERR_STEP_REPORTED_FAILURE: step '{step.name}' reported passed=false: {str(detail)[:300]}"
```

A step whose *deliverable* is data — e.g. "summarize the test run" returning `{"tests": 10, "passed": false}` — is force-failed even though that is the correct content, not a self-report. This is a RONDO-407 v1.1 behavior layered onto the engine; it is not in any REQ-114 MUST, it is applied to all pipelines, and there is no per-step way to disable it (unlike `allow_broad` for the scope guard). For a "prompt program" engine this is a real semantic-overreach footgun.

### 6. LOW-MED — `_resolve_prompt` failed-ref guard ignores the placeholder domain (false-positive abort)
The guard iterates all placeholders but discards whether each is an `inputs.` or `steps.` ref:

```239:243:src/rondo/pipeline.py
    for _domain, key in _placeholders(step.prompt):
        if (key in outputs) and not outputs[key][1]:
            raise PipelineError(
                f"step '{step.name}' references step '{key}' which FAILED — refusing to substitute its output"
            )
```

`outputs` is keyed by step name, but `key` can be an *input* name. If a pipeline has an input named `build` and an earlier step `build` that failed under `on_fail: continue`, a later step using `{{inputs.build}}` is aborted with a spurious "references step 'build' which FAILED." Safe-direction (it over-blocks), but it is a latent correctness bug in the anti-lying core, and no test covers the name-collision case.

### 8. LOW — the contract gate is last-JSON-only while the self-report gate is fail-closed (asymmetry)
`_contract_error` checks only the last JSON object via `extract_json_object` (`pipeline.py:267` → `verify.py:200-215` returns `last`), whereas `_self_reported_failure` scans *all* objects (`pipeline.py:415`, the F3 fix). So a model that emits a real object missing the required key, then appends `{"required_key": ...}`, satisfies the contract on the appended object — the exact append-to-bury pattern F3 hardened against, left open on the contract path. Arguably spec-compliant (req 005 says "the house result-JSON extractor"), so I rank it LOW, but it is an inconsistency worth closing for parity.

### 9. LOW — `--plan` requires every `{{inputs.X}}` to already be supplied

### 10. LOW — plan↔apply estimate divergence is silent
Tied to finding 1 but distinct as an honesty point: nothing links the plan's `within_budget_estimate: false` verdict to apply-time behavior, and the run envelope omits `budget_usd` and any estimate, so a caller has no in-band signal that an apply blew past what the plan predicted. The "Terraform for prompts" promise is that plan and apply agree; here they can disagree with no surfaced warning.

## What is genuinely well-built (REQ-114/115 area)

- **The anti-lying trio is real and tested at the right altitude.** Single-pass placeholder substitution kills the input→step-output re-expansion smuggle (`pipeline.py:232-251`, `test_pipeline_security_review.py::test_f2_*`); the fail-closed all-objects scan defeats appended fake-success (`pipeline.py:383-419`, `test_lie_traps.py`); `dispatch_id` path-traversal is rejected before any `.verifyspec.json` load (`verify.py:218-250`, `test_f1_*`). These are honest, reproduced, regression-pinned fixes.
- **`verify.py` is disciplined defense-in-depth.** `shell=False` argv-only with the string-rejection at validation (`verify.py:55-57`), stdout tail capped *and* sanitized before persistence (`verify.py:122-125`), full-digest sha256 for tamper detection, and the verify block persisted unscrubbed precisely because `[PATH]` redaction corrupted argv — a thoughtful, documented trade-off.
- **The mutation-kill suite is the strongest test asset in the repo.** `test_pipeline_mutation_kills.py` computes asserted values from live code, documents its surviving equivalents honestly, and explicitly builds an *unmocked* contract test for `_default_dispatch` (the mocked seam) — the "one unmocked contract test per mocked seam" rule actually honored, not claimed.
- **No-mocks injected `dispatch=` seam** keeps the engine tests hermetic without faking rondo internals, and the lie-trap suite proves the end-to-end honesty loop survives a mid-stream lie.

## Verdict

The pipeline engine's *honesty* machinery (anti-lying, verify, wiring) is genuinely strong and well-tested. Its *budget* guarantee — the other half of the "Terraform for prompts" thesis — is the weak load-bearing wall: the "hard ceiling" is soft on the first/single step (finding 1), and a MUST envelope field (`duration`, finding 2) is silently unbuilt despite a BUILT/verified spec stamp. Those two, plus the unconditional `passed=false` gate (3), are what I'd block a "REQ-114 is done" claim on.
